1 Code splitting (tách bundle JS) Impact cao✓ DONE

vite.config.js
Vấn đề Khi build, Vite gom tất cả vào 1 file JS (~2MB+). Mỗi lần user mở trang phải tải hết file đó. Tệ hơn: nếu bạn sửa 1 dòng trong App.jsx, file đó thay đổi → browser phải tải lại toàn bộ 2MB, kể cả Three.js không hề thay đổi.
Cách hoạt động sau khi sửa Tách ra 4 file riêng: react-vendor.js · three-vendor.js · r3f-vendor.js · index.js (app code). Mỗi file có hash riêng. Khi bạn sửa app, chỉ index.js thay đổi hash — browser tải lại file nhỏ đó thôi, 3 file vendor được cache sẵn từ lần trước.
Trước
  • 1 file JS duy nhất ~2MB+
  • Sửa 1 dòng app → tải lại toàn bộ
  • Không có build config
Sau — build thực tế
  • three-vendor.js — 734 KB
  • r3f-vendor.js — 665 KB
  • index.js (app code) — 14.9 KB
  • Sửa app chỉ bust index.js
build: {
  target: 'esnext',           // output nhỏ hơn, bỏ polyfill không cần cho browser hiện đại
  chunkSizeWarningLimit: 2000, // tắt cảnh báo vô nghĩa với chunk Three.js to
  rollupOptions: {
    output: {
      manualChunks: {
        'react-vendor': ['react', 'react-dom'],
        'three-vendor': ['three'],
        'r3f-vendor': ['@react-three/fiber', '@react-three/drei', '@react-three/xr'],
      },
    },
  },
},

2 Script nén ảnh bằng Sharp Impact cao✓ DONE

scripts/optimize-images.js (file mới)
Vấn đề Ảnh nằm trong /public/. Vite không xử lý gì với file trong thư mục này — copy nguyên si vào dist/. Dù ảnh nặng 200KB hay 20KB, Vite cũng mặc kệ.
Sharp là gì? Là thư viện nén/resize ảnh chạy bằng Node.js. Đã có sẵn trong devDependencies của dự án (ai đó cài rồi nhưng chưa dùng). Script sẽ đọc từng ảnh WebP, nén chặt hơn với quality 82, resize nếu ảnh quá rộng → ảnh nhỏ hơn 20–30% mà mắt thường khó phân biệt sự khác biệt.
  • Đọc tất cả *.webp từ public/character_image/
  • Resize về max 900px nếu ảnh rộng hơn (đủ cho card + info page)
  • Re-encode WebP với quality: 82, effort: 6
  • In ra size trước/sau để kiểm tra kết quả
Trước (20 ảnh)

  • Nhỏ nhất: 66 KB
  • Lớn nhất: 150 KB
Sau — chạy thực tế
  • Tổng: 1,218 KB (−35.7%)
  • Nhỏ nhất: 43 KB
  • Lớn nhất: 101 KB

3 Tự động hóa script nén ảnh (prebuild) Medium✓ DONE

package.json
Cơ chế npm lifecycle npm có quy tắc: nếu bạn định nghĩa key "prebuild", npm tự động chạy nó trước khi chạy "build" — không cần nhớ chạy thủ công. Tương tự "pretest" chạy trước "test". 
"prebuild": "node scripts/optimize-images.js"

Luồng: npm run build → npm tự chạy optimize-images → Vite build. Không ảnh hưởng npm run dev.

4 Preload ảnh đầu tiên trong index.html Medium✓ DONE

index.html
Vấn đề Trình tự tải trang hiện tại: browser tải HTML → tải JS → JS chạy → React render → browser thấy <img src="..."> → mới bắt đầu tải ảnh. Ảnh đầu tiên phải đợi qua 4 bước trung gian.
Preload là gì? Thêm 1 dòng vào index.html báo browser ngay khi đọc HTML: "Tao biết mày sẽ cần cái ảnh này, tải ngay đi". Browser tải ảnh song song ngay từ đầu, không cần đợi JS chạy xong. 
<link rel="preload" as="image"
      href="/character_image/Untitled_Artwork%201.webp"
      type="image/webp" />

5 fetchPriority cho card đầu tiên Low✓ DONE

src/App.jsx CharacterCard ~line 333
Vấn đề Tất cả 20 ảnh đều có loading="lazy" và browser xếp chúng cùng hàng chờ, ưu tiên như nhau. Nhưng ảnh đầu tiên thì người dùng nhìn thấy ngay khi mở trang — cần hiện càng sớm càng tốt.
fetchPriority là gì? Là gợi ý cho browser: "Ảnh này quan trọng hơn, xử lý trước". Browser dành băng thông ưu tiên cho nó thay vì chia đều cho cả 20 ảnh. Chỉ áp dụng cho card index 0 — các card còn lại vẫn lazy như cũ. 
<img
    src={character.image}
    alt={character.name}
    loading="lazy"
    fetchPriority={index === 0 ? "high" : "auto"}
    onLoad={() => setLoaded(true)}
/>

6 XR Emulator chunks — điều tra kết quả ✓ Không cần làm

Lo ngại ban đầu Build sinh ra các chunk lớn từ @react-three/xr: emulate.js (1.3 MB), living_room.js (1.5 MB), music_room.js (2 MB), office_large.js, office_small.js, meeting_room.js. Nghi ngờ user phải tải về toàn bộ khi mở trang.
Kết quả sau điều tra — không phải vấn đề thật Kiểm tra dist/index.html: chỉ có 3 file được load lúc khởi động: index.js (14.9 KB), three-vendor.js, r3f-vendor.js. Các chunk XR không xuất hiện ở đây.

Kiểm tra bên trong r3f-vendor.js: emulate.js được import bằng dynamic import() và được guard bởi điều kiện:

u = o?.emulate ?? "metaQuest3"
if (typeof window < "u" && u != false) { /* load emulator */ }

Vì store được tạo với emulate: false, biến u = false, điều kiện false !== false = false → block load emulator bị skip hoàn toàn. Các chunk môi trường (living_room, music_room...) cũng lazy và chỉ load khi emulator thực sự khởi động.

Kết luận: browser không bao giờ tải các chunk này. Chúng chỉ nằm trong dist/ nhưng không được request.

? Kiểm tra sau khi làm xong

  1. Chạy npm run build → xem terminal: thấy three-vendor-*.js và r3f-vendor-*.js xuất hiện riêng biệt
  2. Xem log bước prebuild → ảnh giảm ~35% là đúng
  3. Chạy npm run preview → F12 → tab Network → lọc Img → ảnh #1 có Priority High
  4. Reload lại trang → các file vendor-*.js trả về 304 (cache hit, không tải lại)
